## 1. 概要

### 目的
Google Workspace for Education Plus 環境で、**複数チームがリアルタイムに対戦できるクイズすごろくWebアプリ**を構築する。先生はゲームコードを発行し、生徒が各自のデバイスからコード入力でゲームに参加する。

### 主要要件
- ✅ 無料で運用可能
- ✅ 数十人同時アクセス対応（リアルタイム同期）
- ✅ 1人プレイとチーム協力プレイの両対応
- ✅ **Excelファイル編集だけで問題・マスを管理**（管理画面からアップロード → DB取り込み）
- ✅ 教室内で複数クラスが同時開催してもゲームが混ざらない

### ゲームの仕組み
1. サイコロを振ってマスを進む
2. 止まったマスのクイズに答える
3. 正解／不正解でアクション発動（進む・戻る・スキップ券獲得・もう一度・1回休み・スタートへ戻る 等）
4. 全チームがゴールするまで継続（最初の1人で終了モードも選択可）

## 2. 技術スタック

| レイヤー | 技術 | 選定理由 |
|---------|------|---------|
| フロントエンド | Next.js 15 (App Router) | SchoolWorksと同じスタック |
| UI | React + Tailwind CSS | 標準構成 |
| バックエンド | Supabase (無料プラン) | リアルタイム同期・DB・認証を一括提供 |
| ホスティング | Cloudflare Pages | 無料・高速・SchoolWorksと同じ |
| Excel解析 | `xlsx`（SheetJS） | クライアントサイドで.xlsxを解析 |
| 認証（先生） | Supabase Auth (Email/Google) | 管理画面用 |
| 認証（生徒） | 不要（ゲームコード方式） | 登録なしで参加可能 |

### なぜGASではなくSupabaseか
- GASは WebSocket リアルタイム同期を持たない（ポーリングで2-5秒遅延）
- GASのWebアプリは同時実行30件の制限あり（数十人同時で破綻リスク）
- Supabaseは WebSocket で瞬時に同期、無料枠でも実質無制限接続

### 無料枠で収まる根拠
- Supabase 無料：500MB DB、5GB egress、実質無制限接続（月50,000MAUまで）
- Cloudflare Pages 無料：無制限リクエスト、500ビルド/月
- 1週間アクセスがないとSupabaseプロジェクトは一時停止 → 週次でアクセスする運用、もしくは軽微なping機構で対応

## 3. データベース設計

### マスター系テーブル（問題・マス・アクション）
バージョン管理のため `game_sets` で束ねる構造。複数の「ゲームセット」を切り替え可能。

| テーブル | 役割 |
|---------|------|
| `game_sets` | ゲームセットのメタ情報（タイトル、サイコロ設定など） |
| `cells` | マスの定義（マス番号、種別、問題ID、アクションID） |
| `quizzes` | 問題（問題文、4択、正解、解説） |
| `actions` | アクション（タイプ、値、メッセージ） |

### ゲーム系テーブル（リアルタイム同期対象）
| テーブル | 役割 |
|---------|------|
| `game_sessions` | 1回のゲーム（ゲームコード、現在ターン、状態） |
| `teams` | 参加チーム（位置、正解数、スキップ券等） |
| `turn_events` | ターン履歴（イベントログ、リプレイ用） |

詳細なスキーマは `supabase/migrations/20260422_001_initial_schema.sql` を参照。

```
【先生】
  /admin にログイン
  → Excelファイルを編集
  → /admin/import でアップロード → DBへ取り込み
  → /admin/create-game でゲーム作成
  → 発行されたコード/QRを /host/[code] で表示

【生徒】
  / でコード入力 or QRスキャン
  → /join/[code] でチーム名入力
  → /play/[code] で待機（他チームの参加を待つ）
  → 先生が開始 → ゲーム進行（リアルタイム同期）
  → /result/[code] で結果確認
```

## 5. Excelインポート仕様

### 入力ファイル
`すごろくマスター管理テンプレート.xlsx`（リポジトリ同梱、配布用）

### シート構成

Excelテンプレートに含まれるシートは次の5つ。このうち「はじめに」は取り込み対象外（説明用）。

| シート名 | DBテーブル | 備考 |
|---------|-----------|------|
| はじめに | — | 使い方説明（取り込み対象外） |
| ゲーム設定 | `game_sets` | ゲームタイトル、サイコロ設定など |
| マス設定 | `cells` | マス番号0から始まる連番 |
| 問題 | `quizzes` | quiz_code（Q001等）を記入 |
| アクション | `actions` | action_code（ACT_XXX等）を記入 |

### シートごとの列仕様

#### ゲーム設定シート（キー・バリュー形式）
| 設定キー | 例 | DBカラム |
|---------|-----|---------|
| ゲームタイトル | クイズすごろく | `game_sets.name` |
| プレイモード | 1人プレイ＋チーム | （メタ情報） |
| サイコロの面数 | 6 | `game_sets.dice_sides` |
| サイコロ個数 | 1 | `game_sets.dice_count` |
| 回答制限時間（秒） | 30 | `game_sets.answer_time_limit` |
| 説明 | 任意 | `game_sets.description` |

#### マス設定シート
列：マス番号／マス種別／表示テキスト／問題ID／正解時アクションID／不正解時アクションID／有効フラグ／メモ

#### 問題シート
列：問題ID／カテゴリ／難易度／問題文／選択肢A／選択肢B／選択肢C／選択肢D／正解／解説／有効フラグ

#### アクションシート
列：アクションID／タイプ／値／表示メッセージ／有効フラグ／説明

### インポート処理フロー
```
1. ユーザーが .xlsx をアップロード
2. クライアントサイドで xlsx パッケージを使ってJSONに変換
3. バリデーション：
   - 必須シートの存在確認（ゲーム設定／マス設定／問題／アクション）
   - ヘッダー名の一致確認
   - 問題ID・アクションIDの整合性確認（マス設定で参照されているIDが存在するか）
4. プレビュー表示（何件登録されるか、警告があるか）
5. ユーザーが「インポート実行」をクリック
6. Supabase にトランザクションで一括INSERT
   - game_sets → quizzes → actions → cells の順
   - cells の外部キー（quiz_id, action_id）は quiz_code / action_code から解決
7. 完了通知、新規 game_set_id を返却
```

### バリデーションルール
- 問題IDの重複禁止（同一 game_set 内で）
- アクションIDの重複禁止
- マス番号の重複禁止、0から連番
- マス設定で参照される問題ID・アクションIDが存在すること
- マス種別・アクションタイプが定義された値であること
- 正解は A/B/C/D のいずれか

### シート名の揺らぎへの対策
シート名は厳密一致（「マス設定」等）。揺らぎがあった場合はインポート時にエラーとして明示し、配布したテンプレートを必ず使う運用とする。

## 6. リアルタイム同期の仕組み

### Supabase Realtime 使用箇所

```typescript
// チーム全員の画面で監視
const channel = supabase
  .channel(`game:${gameCode}`)
  .on('postgres_changes', { 
    event: '*', 
    schema: 'public', 
    table: 'game_sessions',
    filter: `game_code=eq.${gameCode}`
  }, (payload) => {
    // 現在のターン、状態が変わったら画面更新
  })
  .on('postgres_changes', { 
    event: '*', 
    schema: 'public', 
    table: 'teams',
    filter: `game_session_id=eq.${sessionId}`
  }, (payload) => {
    // チーム位置、正解数、スキップ券などが変わったら画面更新
  })
  .on('postgres_changes', { 
    event: 'INSERT', 
    schema: 'public', 
    table: 'turn_events',
    filter: `game_session_id=eq.${sessionId}`
  }, (payload) => {
    // 新しいターンイベント（サイコロ、移動、回答）をアニメーション表示
  })
  .subscribe();
```

### ターン制御ロジック
1. 現在ターンのチームだけが「サイコロを振る」ボタン有効
2. サイコロ → `turn_events` に INSERT、`teams` の位置を UPDATE
3. 全員の画面にアニメーション再生
4. クイズ表示 → 代表者が回答 → 結果を `turn_events` に記録
5. アクション実行 → `teams` を UPDATE
6. 次のチームへターン移譲：`game_sessions.current_turn_team_id` を UPDATE

### 競合対策
- 現在ターンチーム以外の「サイコロを振る」操作は UI とサーバー両方でブロック
- 同一チームの代表者が複数デバイスから同時に回答した場合は、先着の1件のみ有効（Supabase の upsert + 条件付き UPDATE で実現）

## 7. セキュリティ方針

### RLS (Row Level Security)
- **読み取り**：誰でも可能（ゲームコードを知っていれば参加できる設計）
- **書き込み**：
  - マスター系は認証ユーザー（先生）のみ
  - ゲーム系は anon で可能（プロトタイプ段階）
  - 本番化時には「ゲームコード認証トークン」を導入予定

### ゲームコード
- 6桁、紛らわしい文字（0/O、1/I/L）を除外
- 24時間で自動無効化（`expires_at`）
- `finished` 状態のセッションは再利用可能

### 想定脅威と対策
| 脅威 | 対策 |
|------|------|
| いたずらで他ゲームに参加 | ゲームコードを推測困難に（6桁で約10億通り） |
| 他チームを操作 | クライアント側でターン制御、サーバー側で条件付きUPDATE |
| 大量のセッション作成攻撃 | Rate Limit（Supabase の Edge Functions 経由） |
| Excel編集の事故 | インポート時に差分プレビュー表示 |

## 10. 開発運用ルール

SchoolWorksとは別プロジェクトだが、開発規律は踏襲する。

- **ブランチ戦略**：`main`（将来の本番用） / `develop`（開発の統合先） / `feature/*`（個別機能）
- **直接pushの禁止**：`main` と `develop` には直接pushしない。必ずfeatureブランチ → Draft PR経由
- **コミットメッセージ**：`feat:`, `fix:`, `docs:`, `chore:` 等のプレフィックスを使う
- **Supabaseマイグレーション**：SQLを `supabase/migrations/` に置き、`apply_migration` で適用

## 11. リスクと未決事項

### リスク
| リスク | 影響 | 軽減策 |
|------|------|--------|
| Supabase無料枠の逼迫 | 高 | 定期的にUsage確認、必要ならPro($25/月)へ |
| 7日間無アクセスで一時停止 | 中 | 週次で何らかのアクセスを発生させる（または授業頻度次第） |
| 実装セッションが多すぎる | 中 | SchoolWorks本業を優先、無理せず段階的に |
| 多数決回答モードの複雑性 | 低 | まず代表者モードのみ実装、後日追加 |

### 未決事項（後日決めればOK）
- チーム色・絵文字の具体的な候補
- デザインの方向性（プロトタイプのレトロ調を踏襲するか変えるか）
- 効果音・BGMの有無
- ゲームの勝敗判定ルール詳細（最速到達 / 正解数優先 / 複合）
- 終了後の結果共有（リンクでシェアなど）
